## 2. The additions list

**P38-01. Division-aware tenant model. Priority 1, build queue.** Garda is seven divisions sharing one centralized claims team; RTW help differs by division (aviation has an accommodations team; elsewhere it is operations managers and directors) and by region. The employer tenant needs a division layer (division as sub-organization with its own duty libraries, contacts, and reporting), and the pilot configuration is one division in Ontario. The Prompt 37 onboarding gains a division step for multi-division employers.

**P38-02. Hybrid role access. Priority 1, build queue.** Angelina said her team falls into both Employer and HSSE seats, and multi-role access was promised in the room. Add role composition: one account holding multiple seat permissions with a clean switcher, honoring each seat's walls while switched into it. This also resolves the analytics question cleanly (see section 3).

**P38-03. Objective-first check-in configuration. Priority 1, build queue.** Angelina's strongest single request: no subjective pain scores in what her team sees, because workers will all put ten. Make the check-in question set configurable per tenant, with an objective-first preset (mobility, function, duty completion, check-in streaks) and pain kept as a clinical-side signal only: pain still drives the pain-8 escalation to the clinician, but it never surfaces on the employer or claims views. Our walls already forbid pain scores on the functional-only employer dashboard, so on that surface her request and our law are the same sentence. One honest note for the follow-up: the HSE coordinator seat shows pain today by design (it reads scores for operational need), and Angelina's team holds the HSE half of a hybrid seat (P38-02). So the objective-first preset is a real build that strips pain from her team's coordinator view for this tenant, not an existing absence; make it a visible configuration, not just a claim that it is already handled.

**P38-04. MSK eligibility scope. Priority 1, build queue.** Pilot intake needs an MSK filter: case creation carries an injury category, MSK categories are in scope, everything else politely routed to the existing process during the pilot. Concussion and psychological injuries recorded as explicit Phase 2 candidates (Angelina volunteered they see many, mostly from assaults).

**P38-05. Post-Form-7 intake trigger. Priority 1, build queue.** The confirmed flow: manager completes the 17 page workplace injury package (first four pages are the accident investigation), emails it to Claims Canada within 24 hours, claims team files the employer report, and Continuum picks up at submission. Case creation should therefore accept "board report filed" as its trigger state, with claim number and filing date as first-class fields at intake, not later additions.

**P38-06. Per-worker document archive. Priority 1, build queue.** Angelina asked directly where documents live, and confirmed document storage is necessary. Each worker file needs an archive: the workplace injury package, functional abilities forms, physician reports, the recovery plan, and appointment schedule, uploadable by the claims team, visible by role, clinical content staying behind the wall. Honest scope: the demo represented this concept (a recovery plan shown as attached to the worker file, in narration and a status marker), but the uploadable, role-gated archive itself is net-new. So the build must deliver in real what the room saw represented, and nobody should describe the archive to Garda as already built.

**P38-07. Universal modified duties offer. Priority 2, build queue.** Garda's package includes a standard offer made to every injured worker at claim start regardless of injury: online training modules, completing sales leads, office duties, with a signed accept or decline. Model this as a day-zero universal offer distinct from site-specific duty matching: offer content per tenant, acceptance or decline captured with a timestamp, feeding the claims record.

**P38-08. Division-specific duty libraries. Priority 2, build queue plus Craig.** Craig told the room the division-broken duty lists (aviation, warehousing, everything) are already built and in the database. Our canon library is 209 pre-matched duties across 26 categories, general purpose. Close the gap deliberately: an expansion project producing per-division Garda libraries, aviation carrying airport class constraints Angelina flagged, and the claim routed back to Craig for what already exists on his side versus what needs building. Until reconciled, nobody repeats "thousands of duties, already in the database" to Garda.

**P38-09. Check-in lapse nudges. Priority 2, build queue.** Promised in the room and cheap to honor: if a worker has not checked in for a set number of days, the claims or HSSE dashboard raises a nudge prompt (give this worker a call), with email notification. Functional content only (has not checked in), no health content, worker surface untouched and never shamed: the nudge goes to the team, not the worker, because Angelina's team wants to make the calls themselves.

**P38-10. Physician summary view. Priority 2, build queue.** External treating physicians will not adopt a portal, and Simon said so honestly. What was promised is lighter: a read-only recovery summary a doctor can view. Build it as a share-safe summary (clinician-approved content only, no login friction, expiring link), and treat deeper doctor participation as the Nexus-affiliated path.

**P38-11. White-label theming. Priority 2, build queue.** Promised twice: the platform can be white-labeled so workers feel they are signing into a Garda system. Add tenant theming (name, logo, colors on the worker app and login) with Continuum retained in the footer line. This also serves every future employer.

**P38-12. Alberta portal-assist ingestion. Priority 2, build queue, honesty-scoped.** The room was told information can be pulled from the Alberta portal, including physician reports ("I believe so"). Our verified matrix says no Canadian board publishes a retrieval API. The honest build: Alberta portal-assist ingestion, where a person downloads from the myWCB portal (Angelina confirmed Alberta posts everything) and the platform files it to the right worker automatically. That delivers the value Angelina wanted (documents landing in the right file, notification when new material arrives) without claiming machine retrieval that does not exist. Ontario stays manual, which Angelina herself confirmed ("Ontario does not work").

**P38-13. Nexus Health handoff seam. Priority 3, build queue plus Simon.** The Nexus flow (injury report, telehealth consult with modified duties input, then recovery) sits upstream of Continuum. Garda is not a Nexus customer today, so the seam ships as a documented integration point, not a dependency: Continuum works with any doctor, and lights up further when Nexus is present.

**P38-14. Pilot metering and live ROI view. Priority 2, build queue, ties to the Executive Dashboard queue item.** Pilot pricing is cost per event; the long term model is cost per employee plus cost per event; no user cap. Cost per employee is a coverage-scaled fee against the population served, not a per-login seat charge, so it holds the no-per-user-fee and no-user-cap commitments; the follow-up should state that distinction plainly, because Angelina asks precise follow-ups. Add event metering (a countable definition of event, proposed: an accepted MSK case entering coordination) and the live pilot view Craig promised ("day 15 or day 50"): cases in coordination, check-in completion, days to first safe duties against baseline, so Angelina and Andree-Anne watch value accrue in real time.

**P38-15. Consent alignment. Priority 1, counsel.** Garda recently added a consent form to their injury package allowing direct contact with treating doctors. Our worker consent (12j) and the pilot consent text (already the standing counsel item) must be reconciled with Garda's form so a worker is not consented twice in conflicting words. Send both to counsel together.

**P38-16. Naming, corrected and pending. Priority 3, ledger.** The company is GardaWorld (the transcript's "Garta" is a transcription error; use Garda in speech, GardaWorld in documents). Angelina said only "Sigma," so the SIGMA-HR versus SIGMA-RH confirmation remains open; it now belongs in the follow-up to Angelina rather than a room question, since she is the named contact.
